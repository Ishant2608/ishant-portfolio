Zenska.ph

Multi-vendor E-commerce Platform

Timeline 2026 — Present
Status Live in Production

02Overview

Zenska.ph is a multi-vendor e-commerce marketplace — multiple independent sellers operating storefronts on one platform, each with their own product catalog, orders, and dashboard.

Unlike a single-store WooCommerce build, a marketplace needs three distinct user experiences sharing one underlying system: an admin layer to oversee the whole platform, a vendor layer for sellers to manage their own business, and a customer layer for the actual shopping experience. I worked on all three.

03The Problem

A standard WordPress/WooCommerce setup is built around one store, one admin, one set of products. Zenska.ph needed many independent vendors operating simultaneously without stepping on each other's data — and without the platform turning into an operational bottleneck for the team running it.

  • Vendors needed to self-manage products and orders without admin intervention for every action
  • Onboarding new vendors manually didn't scale past a handful of sellers
  • Admin, vendor and customer roles needed clearly separated permissions and views
  • The platform had to stay performant as catalog and vendor count grew

04The Solution

I built three role-specific dashboards on a shared codebase, backed by a guided vendor onboarding flow and a REST API layer connecting the React frontend to WordPress as the underlying CMS/data layer.

  • Admin Dashboard — platform-wide oversight: vendor approvals, order monitoring, catalog moderation
  • Vendor Dashboard — self-service product management, order fulfillment, and store settings per seller
  • Customer Dashboard — order history, account management, and the shopping experience itself
  • Guided onboarding — a validated, step-by-step flow that gets new vendors live without manual admin setup

06My Role

Software Engineer — working as part of the team building Zenska.ph end-to-end, with primary ownership of the dashboard experiences and the API integration layer connecting them to WordPress.

  • Built the Admin, Vendor and Customer dashboard UIs in React and TypeScript
  • Designed and implemented the vendor onboarding flow
  • Handled API integration and core business logic between frontend and WordPress
  • Collaborated on architecture decisions for how the three roles share underlying data

07Design Approach

Three roles, one visual language — but each dashboard tuned to what that user actually needs to see first.

Admins needed density and oversight: tables, filters, and approval queues front and center. Vendors needed clarity and confidence: a small number of clear actions (add product, view orders, check payouts) without the noise of platform-wide data they don't need. Customers needed the dashboard to feel like a normal store account — minimal, familiar, out of the way of actually shopping. Keeping the underlying component system consistent across all three kept the build maintainable without making every screen feel identical.

08Development Process

01

API & data modeling

Mapped how vendor, product, order, and customer data needed to relate before writing UI — got this wrong once early on and it was cheaper to fix before the dashboards existed.

02

Component-first build

Built shared React components (tables, forms, cards) once, then composed each dashboard from them — kept the three experiences visually consistent without three separate implementations.

03

Vendor onboarding flow

Built and validated step-by-step, testing with realistic vendor data at each stage rather than building the whole flow before testing any of it.

04

Iteration in production

As a live platform, changes ship incrementally with real vendor and order data in play — so changes are tested carefully before reaching production.

09Performance & Impact

Performance Improvements

  • API calls scoped per-role so each dashboard only fetches the data it actually needs
  • Component reuse across dashboards reduced duplicate code and bundle weight
  • Hosted on Railway for straightforward deploys and environment management

Business Impact

  • Vendors can onboard and manage their own stores without manual admin setup for every seller
  • Admin team gets platform-wide visibility without digging through raw data
  • A genuine multi-vendor marketplace, not a single-store site pretending to be one
